## 1. 产品一句话定位

AI 漫剧生成器是一款 PC 端 Web 创作工作台，面向普通创作者、短视频作者、漫画/网文爱好者，通过「角色设定 → 剧情输入 → 分镜编排 → AI 视频生成 → 后期导出」的流程，将文字故事转化为角色一致、镜头连贯、有漫剧风格的短视频。

产品的核心不是单一 AI 生成器，而是：

> 一个模型中立、用户自带 API Key、围绕漫剧创作流程组织的 AI 编排平台。

## 2. 核心产品判断

### 2.1 原方案中应保留的核心

ClaudeCode 原方案的优势是完整，已经覆盖了从产品定位、UI/UX、技术架构、模型管理到 API Key 管理的主要模块。以下方向应继续保留：

1. 模型中立
   - 不绑定单一模型。
   - 用户可以在 LLM、图片、视频、TTS、音乐、音效等环节自由选择模型。
   - 用户使用自己的 API Key，平台作为编排层。

2. 端到端创作流程
   - 角色设计。
   - 故事编排。
   - 分镜生成。
   - 镜头提示词控制。
   - 视频生成。
   - 配音、字幕、音乐、导出。

3. 角色一致性
   - 这是 AI 漫剧区别于普通 AI 视频工具的关键卖点。
   - 用户不只是生成单条视频，而是希望同一个角色能跨镜头、跨章节、跨集保持稳定。

4. 镜头语言引导
   - 普通用户不会写专业镜头提示词。
   - 产品必须把景别、角度、运镜、光影、情绪这些专业概念转成可选择、可理解、可编辑的交互。

5. 暗色影院感 + 分镜网格的 UI 方向
   - 这个视觉方向符合产品主题。
   - 分镜网格可以成为产品的视觉记忆点。

### 2.2 Codex 建议中应采纳的核心

Codex 的建议主要解决“原方案太重、太长、太专业”的问题，应重点采纳：

1. 默认快速路径
   - 原 8 步流程对新手过重。
   - 产品冷启动阶段必须让用户尽快看到第一个结果。
   - 应提供「极速模式」作为默认入口，让用户 3-4 步看到粗剪成片。

2. 分镜与镜头控制合并
   - 分镜天然包含镜头描述。
   - 不应拆成两个页面让用户理解两套概念。
   - 应在同一个分镜编辑页中，通过卡片展开编辑镜头参数。

3. 配音/字幕时间轴应前置
   - 视频生成后再配音，容易出现时长不匹配。
   - 应先确定分镜时长、台词、旁白、字幕，再生成视频片段。
   - 视频生成应服务于已经确定的叙事节奏。

4. 模型商店与 API Key 配置合并
   - 用户的真实路径是“我想用某个模型 → 我去配置这个模型的 Key”。
   - 模型详情页就应该能直接配置、验证、保存 Key。
   - 不应让用户在模型商店、设置页、项目页之间来回跳。

5. 技术 Adapter 按能力拆分
   - LLM、图像、视频、TTS、音乐的输入输出差异极大。
   - 不应强行用一个统一 `ModelAdapter` 承接所有能力。
   - 应拆成 `LLMAdapter`、`ImageAdapter`、`VideoAdapter`、`TTSAdapter`、`MusicAdapter`、`SoundAdapter`。

6. 文件链路区分素材与产物
   - 用户上传素材：前端直传对象存储。
   - AI 生成产物：由后端中转、任务轮询或模型平台回调写入对象存储。
   - 这两条链路不能混在一起设计。

## 3. 整合后的产品形态

### 3.1 产品名称建议

当前可暂定：

- AI 漫剧生成器
- AI 漫剧创作台
- 漫剧导演台
- Storyboard AI Studio
- MangaDrama Studio

如果做中文市场，建议优先使用：

> AI 漫剧创作台

原因：
- “生成器”听起来像单次工具。
- “创作台”更符合多步骤、多模型、多素材、多项目的工作流定位。

### 3.2 产品核心体验

用户进入产品后，不应该先感受到复杂的专业流程，而应该感受到：

> 我只要给出角色和故事，系统就能先帮我生成一个能看的漫剧草稿；如果我想精修，再进入专业的分镜、镜头、模型、时间线控制。

因此产品应设计为双模式：

1. 极速模式：默认给新手使用。
2. 专业模式：给进阶用户精调使用。

## 4. 整合后的用户流程

## 4.1 极速模式：默认路径

目标：让用户在 3 分钟内看到第一个粗剪作品。

```text
Step 1：创建/选择角色
    - 输入角色名、外貌、性格
    - 或从角色模板中选择
    - 生成或上传角色参考图

Step 2：输入故事
    - 输入一段剧情、小说片段、短剧梗概
    - 选择风格：校园 / 古风 / 悬疑 / 恋爱 / 热血 / 赛博等

Step 3：一键生成粗剪
    - AI 自动拆分分镜
    - 自动生成静态预览图
    - 自动分配镜头语言
    - 自动生成字幕/旁白草稿
    - 自动调用视频模型生成短片段

Step 4：预览与精修
    - 用户观看粗剪结果
    - 不满意的分镜可单独重写、重画、重生成
    - 可切换到专业模式继续编辑
```

极速模式的关键不是一次生成完美成片，而是快速形成“可观看草稿”。

### 4.2 专业模式：完整创作路径

目标：给认真创作的用户提供可控、可回退、可复用的工作流。

```text
1. 项目初始化
   ↓
2. 角色与角色变体管理
   ↓
3. 故事 / 剧本 / 台词编排
   ↓
4. 分镜 + 镜头控制一体化编辑
   ↓
5. 静态预览 + 故事板阅读预览
   ↓
6. 音频 / 字幕 / 时间轴预排
   ↓
7. 视频片段生成
   ↓
8. 后期合成与导出
```

与原方案相比，整合后的主要变化：

| 原方案 | 整合后 |
|---|---|
| 创建项目时填写名称、风格、比例、时长 | 只要求最少信息，其他配置后置 |
| 分镜生成和镜头控制分两页 | 合并为同一页，分镜卡片展开编辑镜头参数 |
| 视频生成后再做配音/字幕 | 先确定台词、旁白、字幕、时长，再生成视频 |
| 模型商店和 Key 管理分开 | 模型详情内直接配置 Key |
| 线性 8 步向导 | 检查表式导航，允许自由回跳 |

## 5. 核心模块整合说明

## 5.1 项目管理模块

### 定位

项目管理不应成为新手的第一道门槛，而应作为自动保存和作品管理的底层能力。

### MVP 做法

P0：
- 新建项目只需要项目名称，甚至可以自动命名。
- 支持最近项目列表。
- 自动保存每一步内容。
- 支持项目级模型配置。
- 支持导出最终视频和素材包。

P1：
- 项目版本历史。
- 多集管理。
- 项目模板。

暂缓：
- 复杂团队协作。
- 企业审片流。

---

## 5.2 角色设计与一致性模块

### 定位

角色一致性是本产品的核心护城河之一。必须从“单张参考图”升级为“角色资产系统”。

### 整合后的角色结构

```text
角色 Character
├─ 基础信息
│  ├─ 姓名
│  ├─ 年龄
│  ├─ 性别
│  ├─ 身份
│  ├─ 性格
│  └─ 角色描述提示词
│
├─ 主形象 Main Identity
│  ├─ 主参考图
│  ├─ 标准服装
│  ├─ 标准发型
│  └─ 标准脸部特征
│
├─ 角色变体 Variants
│  ├─ 表情变体：开心 / 哭泣 / 愤怒 / 惊讶
│  ├─ 服装变体：校服 / 睡衣 / 古装 / 战斗服
│  ├─ 角度变体：正面 / 侧面 / 背影 / 半身
│  └─ 场景变体：白天 / 夜晚 / 雨天 / 战斗状态
│
└─ 模型适配信息
   ├─ 支持参考图的模型：传参考图
   ├─ 支持角色引用的模型：传 character ref / cref
   ├─ 支持 LoRA 的模型：绑定 LoRA
   └─ 不支持参考图的模型：注入角色提示词
```

### 角色锁定策略

不要只给用户一个“锁定角色”开关，而应给用户“锁定强度”：

| 锁定强度 | 适用场景 | 系统行为 |
|---|---|---|
| 宽松 | 创意发散、不同画风尝试 | 主要注入文字描述，允许造型变化 |
| 中等 | 常规漫剧生成 | 使用参考图 + 角色提示词，兼顾一致性和画面自由度 |
| 严格 | 连载角色、品牌 IP | 强制参考主形象/变体，限制发型、服装、脸部变化 |

### MVP 要求

P0：
- 创建角色。
- 生成/上传主参考图。
- 角色提示词自动生成。
- 角色绑定到分镜。
- 在生成提示词中自动注入角色信息。

P1：
- 角色变体集合。
- 角色锁定强度。
- 跨集角色复用。

P2：
- LoRA / FaceID / InstantID 深度绑定。
- 角色一致性评分。

---

## 5.3 故事与剧本模块

### 定位

故事模块不只是一个大文本框，而是后续分镜、台词、字幕、镜头节奏的来源。

### 需要支持的输入类型

P0：
- 一段剧情梗概。
- 一段小说文本。
- 简单分集剧情。

P1：
- 标准剧本格式。
- 多角色对话。
- 多集连续剧大纲。

### 创作助手统一入口

原文档里存在多个“AI 帮我写”的入口：
- 角色提示词生成器。
- 剧情润色。
- 分镜拆分。
- 提示词优化。
- 关键词推荐。

这些不应让用户分别理解。整合后统一为：

> 创作助手 AI Assistant

它根据用户所在上下文自动判断任务：

| 当前页面 | 用户输入 | 助手行为 |
|---|---|---|
| 角色页 | “她是一个有点忧郁的高中生” | 生成角色描述和外貌提示词 |
| 故事页 | “帮我把这个故事更有反转” | 润色剧情、增强戏剧冲突 |
| 分镜页 | “这一段节奏太慢” | 合并/拆分分镜，调整时长 |
| 镜头参数 | “想要更浪漫一点” | 推荐光影、景别、运镜、色调 |
| 视频生成失败 | “这个画面不像女主” | 建议加强角色参考或切换模型 |

## 5.4 分镜与镜头控制模块

### 定位

这是产品的核心工作区。它应同时承担：
- 剧情拆分。
- 镜头描述。
- 角色绑定。
- 静态预览。
- 视频生成前的确认。

### 页面结构建议

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部：项目名 / 当前模式 / 模型状态 / 保存状态 / 预览 / 导出       │
├───────────────┬─────────────────────────────────────────────┤
│ 左侧检查表导航 │ 主区域：分镜网格                               │
│               │                                             │
│ 角色           │ ┌────────┐ ┌────────┐ ┌────────┐             │
│ 故事           │ │ 分镜1  │ │ 分镜2  │ │ 分镜3  │             │
│ 分镜           │ │ 预览图 │ │ 预览图 │ │ 预览图 │             │
│ 时间轴         │ │ 文案   │ │ 文案   │ │ 文案   │             │
│ 生成           │ └────────┘ └────────┘ └────────┘             │
│ 导出           │                                             │
├───────────────┴─────────────────────────────────────────────┤
│ 底部 / 右侧：选中分镜详情                                    │
│ - 出镜角色                                                     │
│ - 场景描述                                                     │
│ - 台词/旁白                                                    │
│ - 时长                                                         │
│ - 景别 / 角度 / 运镜 / 光影 / 情绪                              │
│ - 完整提示词                                                    │
│ - 快速静态预览 / 重新生成                                      │
└─────────────────────────────────────────────────────────────┘
```

### 必须增加的闭环

P0 必须支持：

1. 分镜静态预览
   - 每个分镜可调用图像模型生成一帧预览图。
   - 用户在生成视频前先检查构图和角色。

2. 分镜展开编辑
   - 分镜卡片默认简洁展示。
   - 点击后展开镜头参数。
   - 不需要跳到另一个“镜头控制页”。

3. 批量镜头设置
   - 可批量选择分镜。
   - 统一设置画风、景别、运镜节奏。

P1 支持：

1. 故事板阅读模式
   - 像看漫画一样连续阅读分镜。
   - 在视频生成前检查叙事节奏。

2. AI 自动分配镜头语言
   - 用户点击一次，AI 为所有分镜分配景别、角度、运镜和情绪。

## 5.5 音频、字幕与时间轴模块

### 定位

时间轴不应只在视频生成后出现。它应在视频生成前就参与节奏设计。

### 整合后顺序

```text
分镜文本
  ↓
台词 / 旁白 / 字幕
  ↓
每个分镜预估时长
  ↓
音频生成或占位
  ↓
视频生成按分镜时长对齐
  ↓
后期合成微调
```

### MVP 做法

P0：
- 每个分镜设置时长。
- 每个分镜绑定字幕/旁白文本。
- 支持基础 TTS 生成配音。
- 导出时合成字幕和音频。

P1：
- 多角色配音。
- 音色与角色绑定。
- 背景音乐。
- 常用音效。

P2：
- 口型同步。
- 音乐自动卡点。
- 音效自动匹配。

---

## 5.6 视频生成模块

### 定位

视频生成不是孤立按钮，而是分镜任务队列。

### 关键能力

P0：
- 单镜生成。
- 批量生成。
- 任务队列。
- 进度展示。
- 失败重试。
- 对失败分镜单独重生成。
- 结果保存到项目素材库。

P1：
- 切换模型重试。
- 首帧/尾帧控制。
- 批量参数覆盖。
- 不同模型生成结果对比。

P2：
- 自动选择最适合模型。
- 成本/速度/质量策略路由。

---

## 5.7 模型与 API Key 管理模块

### 定位

这是产品的能力中枢，但不应该把复杂性过早暴露给新手。

### 整合后的原则

1. 模型商店和 Key 配置合并。
2. 默认只让用户理解“当前项目用哪些模型”。
3. 全局默认配置存在，但弱化展示。
4. 任务级切换属于高级功能。

### 推荐交互

```text
用户在视频生成页选择“可灵 Kling”
    ↓
如果未配置 Key
    ↓
弹出模型详情 / Key 配置面板
    ↓
展示：模型介绍、适用场景、价格说明、官方获取 Key 链接
    ↓
用户粘贴 Key，点击验证
    ↓
验证通过后直接用于当前项目
```

### 配置层级

内部仍然可以保留三层：

```text
任务级配置 > 项目级配置 > 全局默认配置
```

但 UI 上建议：

- P0：只暴露项目级配置。
- P1：在高级设置中暴露全局默认。
- P2：在单任务弹窗中暴露任务级临时切换。

### P0 首批模型建议

为了降低接入成本，首批不应追求覆盖所有模型，应选择少量代表模型打通链路。

| 能力 | 建议首批 | 原因 |
|---|---|---|
| LLM | DeepSeek、OpenAI Compatible | 中文成本低，OpenAI 兼容生态广 |
| 图像 | Replicate FLUX、通义万相或即梦 | 一个国际 API，一个国内 API |
| 视频 | 可灵、豆包/即梦 | 国内用户可用性强 |
| TTS | ElevenLabs、豆包语音或 MiniMax | 覆盖英文和中文 |
| 合成 | FFmpeg | 稳定、成熟、可控 |

## 6. UI/UX 整合说明

## 6.1 视觉方向

保留 ClaudeCode 原 UI 方向：

> 暗色影院感 + 漫画分镜网格 + 可控参数面板。

关键词：

`剧场`、`分镜`、`胶片`、`漫画网点`、`导演台`、`故事板`、`放映机`、`霓虹弱光`

### 推荐色彩

| Token | 色值 | 用途 |
|---|---|---|
| 影院黑 | `#0B0C0F` | 主背景 |
| 深灰板 | `#14161B` | 面板、卡片 |
| 中灰板 | `#1E2128` | 输入区、编辑区 |
| 分割线 | `#2A2E36` | 边框 |
| 漫剧紫 | `#7C5CFF` | 主品牌色 |
| 荧光青 | `#00E5C9` | 进度、选中、成功 |
| 暖光橙 | `#FF9E5E` | 警告、重点 |
| 主文本 | `#F2F3F5` | 主要文字 |
| 次文本 | `#9AA0AC` | 说明文字 |

### Codex 建议修正

原方案整体偏暗，适合展示视频，但不一定适合长时间写故事和改提示词。

因此：

- 视频预览区可以保持影院黑。
- 文本编辑区应使用中灰底，降低视觉疲劳。
- 故事输入、提示词编辑、字幕编辑不应使用纯黑背景。

## 6.2 导航方式

原方案是线性步骤导航。整合后建议改为：

## 7.2 Next.js 与 NestJS 边界

必须明确边界，避免双后端混乱。

建议：

| 层 | 职责 |
|---|---|
| Next.js | 页面渲染、前端状态、表单、用户交互、基础 auth 跳转 |
| NestJS | 所有业务 API、模型调用、任务创建、Key 加密、文件记录、权限控制 |
| Worker | 执行长耗时 AI 任务、轮询第三方结果、处理回调、写入对象存储 |

约定：

- Next.js 不写核心业务 API。
- Next.js API Routes 只保留极少量必要代理或健康检查。
- 所有项目、角色、分镜、模型、任务相关 API 都走 NestJS。

## 7.3 Adapter 设计

不要使用一个大而全的 `ModelAdapter`。应按能力拆分。

```typescript
type CapabilityType = 'llm' | 'image' | 'video' | 'tts' | 'music' | 'sound';

interface BaseAdapter {
  provider: string;
  modelId: string;
  validateKey(apiKey: string): Promise<KeyValidationResult>;
}

interface LLMAdapter extends BaseAdapter {
  generateText(input: LLMInput, config: LLMConfig): Promise<LLMResult>;
}

interface ImageAdapter extends BaseAdapter {
  generateImage(input: ImageInput, config: ImageConfig): Promise<ImageResult>;
}

interface VideoAdapter extends BaseAdapter {
  generateVideo(input: VideoInput, config: VideoConfig): Promise<VideoTaskResult>;
  getVideoTaskStatus(taskId: string): Promise<VideoTaskStatus>;
}

interface TTSAdapter extends BaseAdapter {
  synthesize(input: TTSInput, config: TTSConfig): Promise<AudioResult>;
}
```

上层通过工厂路由：

```typescript
adapterFactory.getAdapter(capability, modelId)
```

这样既保留统一注册，又避免每个 Adapter 内部写大量 capability 判断。

### 7.4.1 用户素材链路

## 7.5 API Key 安全

保留原 PRD 的安全要求，并作为 P0。

P0 必须做到：

- HTTPS 传输。
- API Key 服务端加密存储。
- 使用 AES-256-GCM 或云 KMS。
- 日志永不记录完整 Key。
- 前端永不展示完整 Key。
- 用户只能访问自己的 Key。
- Key 调用必须经过服务端，不暴露给前端。

本地开发可使用环境变量里的主密钥；生产环境应使用云 KMS 或独立密钥管理服务。

## 10. 主要风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 用户流程太重 | 8 步流程容易流失 | 默认极速模式，专业模式后置 |
| 角色一致性难落地 | 不同模型参考图能力差异大 | 分层策略 + 角色变体 + 锁定强度 |
| 视频生成成本高、反馈慢 | 用户反复试错成本高 | 先静态预览，再生成视频 |
| API Key 安全风险 | 用户 Key 泄露会严重影响信任 | 服务端加密、日志脱敏、权限隔离 |
| 多模型适配复杂 | 每个模型参数和任务机制不同 | 按能力拆分 Adapter，先少量模型打通 |
| Next.js/NestJS 边界混乱 | 双层架构可能导致逻辑分散 | 明确 Next 只做 UI，业务 API 全走 NestJS |
| 音画不同步 | 视频后配音会反复返工 | 台词/字幕/时长前置 |
| 纯黑 UI 编辑疲劳 | 长文本输入不舒服 | 编辑区使用中灰背景或编辑模式 |

## 12. 最终结论

ClaudeCode 的原方案内容完整，适合做“产品全景图”；Codex 的建议更接近真实用户使用和工程落地，尤其指出了流程过长、概念过多、反馈闭环不足、技术边界不清的问题。

整合后的方向应该是：

1. 定位不变：模型中立 + 用户自带 Key + AI 漫剧创作编排平台。
2. 默认体验改变：从 8 步专业流程，改为“极速模式优先，专业模式后置”。
3. 核心工作区改变：分镜和镜头控制合并，形成一体化故事板编辑器。
4. 生成顺序改变：先静态预览和音频/字幕时间轴，再生成视频。
5. 模型管理改变：模型商店和 API Key 配置合并，默认只暴露项目级配置。
6. 技术架构细化：Next.js 负责前端，NestJS 负责业务 API；Adapter 按能力拆分；素材和产物链路分开。
7. MVP 收敛：先做 30-60 秒短漫剧闭环，不要一开始追求社区、评分、账单、企业协作等外围功能。

一句话总结：

> 这款产品第一版不应该做成“复杂的 AI 影视工作站”，而应该做成“能让普通用户快速得到第一条漫剧草稿，并允许进阶用户逐镜精修的 AI 漫剧创作台”。
