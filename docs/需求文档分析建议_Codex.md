# AI 漫剧生成器 — 需求文档分析与改进建议

> 分析日期：2026-06-29  
> 分析对象：`产品方案 V1.1`、`UIUX设计规划 V1.0`、`技术选型方案 V1.0`、`模型与APIKey管理模块PRD V1.0`  
> 状态评审人：Codex

---

## 一、核心架构问题

### 1. 8 步流程过长，用户容易在中途放弃

**现状**：创建项目→角色设计→故事编排→分镜生成→镜头控制→视频生成→后期合成→导出发布，共 8 个步骤。

**问题**：对新手来说步骤太多，每一步都有心理门槛。"分镜"和"镜头控制"这类偏专业步骤最容易让非专业用户流失。

**建议**：

- 引入双路径模式
  - **快速模式（极速路径）**：合并为 3-4 步——角色设定 → 剧情输入 → 一键生成粗剪视频 → 精调。让用户 3 分钟内看到第一个作品，建立成就感后再引导深入。
  - **专业模式**：保留当前的 8 步完整流程，面向进阶用户。
- 快速模式作为默认入口，专业模式作为可选切换。

### 2. 分镜生成与镜头控制拆成两步，用户难区分

**现状**：第 4 步"AI 自动生成分镜"和第 5 步"镜头与提示词控制"是独立页面。

**问题**：从用户视角看，分镜（storyboard）天然包含镜头描述，分成两步会产生困惑——"我在第 4 步拿到了什么，跟第 5 步是什么关系？"

**建议**：

- 合并为一个页面：AI 生成分镜列表 → 每个分镜卡片可展开编辑镜头参数。
- 用卡片展开/折叠代替页面跳转，将两层信息组织在一个视图内。

### 3. 后期合成放在视频生成之后，违背实际生产逻辑

**现状**：先生成视频片段 → 再后期合成配音、字幕、音乐 → 再导出。

**问题**：如果生成的视频跟配音时间轴不匹配，必须重新生成视频，回退成本极高。

**建议**：

- 把"配音"提到视频生成之前：让用户先确定配音时长，视频生成时参考时间轴对齐。
- 或者改为：先做音频和字幕（与分镜时间轴对齐）→ 再触发视频生成时直接带上配音和字幕参数。

### 4. 角色一致性锁定方案过于笼统

**现状**：仅表述为"将选定角色图作为参考图，后续生成统一角色形象"。

**问题**：不同模型对参考图的处理方式差异极大（FLUX 用 IP-Adapter/Redux，MJ 用 Cref，SD 用 ControlNet+LoRA），一句话描述掩盖了模型间的实现鸿沟。这个功能是产品的最大价值点之一，但落实路径不清晰。

**建议**：

- 拆分为三层策略：
  - (a) 依赖模型自身的参考图能力（如 MJ Cref）
  - (b) 平台端做图预处理 + 参考图注入
  - (c) 对不支持参考图的模型，退回到提示词级的角色描述注入
- 用户端暴露**角色锁定强度**选择（宽松 / 中等 / 严格），而非简单开关。

## 二、产品体验问题

### 5. 模型商店和 API Key 管理割裂成两个功能点

**现状**：用户需先知道模型商店有某个模型 → 去 Key 管理页面配置 → 再回工作流中选择。

**问题**：路径太绕，用户的思维模型是"我想用可灵 → 配可灵的 Key"，应该在同一个视图中完成。

**建议**：

- 模型详情页内嵌 API Key 配置入口，用户看到模型就能在同一页面完成 Key 配置。
- 不强制用户在 Settings 和 Project 之间来回跳转。

### 6. 全局 / 项目 / 任务三级模型选择增加认知负担

**现状**：优先级链为"任务级 > 项目级 > 全局默认"，用户需要在三个层级管理模型选择。

**问题**：对非专业用户来说太复杂，需要理解三层覆盖逻辑。

**建议**：

- 默认只暴露"项目配置"一层，全局默认作为后台兜底。
- 如果用户未在项目中配置模型，用简洁的引导入口提示他去配置，而不是展示三层优先级让他理解。

### 7. 五层提示词体系有重叠，用户迷惑

**现状**：产品方案设计了"结构化表单" + "自然语言辅助" + "专家模式"三层提示词引导，外加"角色提示词生成器"和"AI 剧情润色"。

**问题**：用户面对 5 种不同的"写文字"入口，不知道它们之间的边界和关系。

**建议**：

- 统一为一个入口：**创作助手（AI Assistant）**。
- 用户描述意图 → AI 理解当前上下文（在第几步、在写角色描述还是分镜提示词）→ 自动选择最适合的交互模式。
- 避免让用户自己做模式匹配。

### 8. 项目管理启动太重

**现状**：新建项目要求用户在第一步选择漫画风格、视频比例、目标时长。

**问题**：新手在创建项目时还没形成这些概念，先选择会形成阻力。

**建议**：

- 新建项目只需要输入名称，其他设置改为可选，在后续流程中按需出现。
- 或者更激进：没有"项目列表"，每次创作就是一个独立页面会话，自动保存，用"最近创作"替代项目管理概念。

## 三、技术方案风险

### 9. 素材与产物两条链路未区分

**现状**：技术选型写到"大文件建议前端直传到对象存储"，但没有区分是用户上传的素材还是 AI 生成的结果。

**问题**：用户上传的素材（角色图、参考图）可以直传，但 AI 生成产物（视频片段）需要后端中转或模型平台回调，两者链路完全不同。

**建议**：

- 明确"素材"和"产物"两条数据处理链路：素材走前端直传，产物走模型回调或平台中转。
- 这个区分影响后端架构设计中文件流的方向。

### 10. Next.js + NestJS 双层架构边界不清晰

**现状**：前端用 Next.js 14（自带 API Routes），后端另起 NestJS 服务。

**问题**：团队容易把部分逻辑写在 Next.js API Routes 里、另一部分写在 NestJS 里，导致维护混乱。

**建议**：

- 明确约定：Next.js 只做渲染层 + 基础的 `/auth` 代理，所有业务 API 走 NestJS。
- 或相反：如果团队人力有限，砍掉 NestJS 层，直接用 Next.js API Routes + 任务队列。

### 11. 通用 Adapter 接口按 capability 拆分

**现状**：方案设计了一个统一的 `ModelAdapter` 接口，通过 `capability` 字段区分能力类型。

**问题**：视频模型和 LLM 的 input/output shape 差异巨大，强行统一会导致 Adapter 实现中充斥着 `if (capability === 'video')` 的分支。

**建议**：

- 按能力环节拆分为多个接口族：`LLMAdapter`、`ImageAdapter`、`VideoAdapter`、`TTSAdapter`、`MusicAdapter`。
- 每个接口族定义自己的 input/output type。
- 上层用工厂模式按 `(capability, modelId)` 路由到具体 Adapter 实例。

## 四、遗漏的需求

### 12. 缺少分镜预览闭环

**现状**：用户要等到视频生成完成才能看到效果。

**问题**：调一个分镜的构图需要经历完整生成队列，试错成本太高。

**建议**：

- 每个分镜卡片增加"快速预览"按钮，调用用户配置的图像模型生成一帧静态预览图。
- 用户在分镜阶段就能看到画面效果，大幅缩小反馈循环。

### 13. 缺少故事板叙事预览模式

**现状**：分镜网格只是表格化展示。

**问题**：用户需要理解"连续看下来讲了个什么故事"，现在缺少类似漫画阅读的"按顺序预览"模式。

**建议**：

- 分镜页增加**翻页预览**模式，将分镜静态图按顺序排列成类似漫画书的阅读体验。
- 用户在生成视频前先"读故事"，发现节奏问题。

### 14. 多角色跨集一致性未覆盖

**现状**：文档提到"支持一剧多角色"和"多集连续创作"，但没有覆盖同一角色跨集的形象一致性。

**问题**：当角色在下一集换服装或造型后，参考图体系如何管理？

**建议**：

- 引入**角色状态变体**概念：角色在不同场景/情绪/服装下有不同的参考图，但都属于同一个角色的变体集合。
- 用户管理的是"角色 → 变体集合"，而不是单张参考图。

### 15. 缺少分镜批量编辑能力

**现状**：只能逐镜编辑。

**问题**：当一部漫剧有 50+ 个分镜时，逐镜编辑不可接受。

**建议**：

- 支持批量选择分镜 → 统一设置镜头参数（景别、运镜风格等）。
- 支持"AI 自动为所有分镜分配镜头语言"的一键操作。

## 五、设计系统建议

### 16. 线性步骤导航不灵活

**现状**：侧边栏导航固定 8 步顺序，用户只能依次前进。

**问题**：真实创作是非线性的——用户可能先设计角色，再写故事，然后回头改角色描述。

**建议**：

- 侧边导航改为"已完成 / 激活 / 待处理"的状态标识。
- 用户可以在已完成的步骤之间自由跳转，更像"创作检查表"而不是"向导进度条"。

### 17. 纯黑背景在编辑场景下视觉疲劳

**现状**：主背景 `#0B0C0F`，所有面板为深灰。

**问题**：对于需要长时间阅读和输入的角色设计页、故事编排页、提示词编辑区，纯黑背景看久了眼睛容易累。

**建议**：

- 文字编辑区域（故事输入框、提示词编辑区）使用中灰色底替代纯黑。
- 或提供"编辑模式"切换，将编辑区背景调亮一级。

## 六、优先级重新排序

以下对照现行 PRD 中的 P0/P1/P2 标注，给出调整建议：

| 功能 | 当前优先级 | 建议 | 原因 |
|------|-----------|------|------|
| 快速模式（3-4 步出片） | 未提及 | **P0** | 削减用户流失的核心手段 |
| 分镜静态预览图 | 未提及 | **P0** | 用户需要即时反馈 |
| 多角色跨集一致性 | 未提及 | **P1** | 连载场景的关键能力 |
| 模型推荐组合 | P1 | **P2** | 前期用户少，推荐无数据 |
| 用量统计和费用估算 | P1 | **P2** | 早期量级不需要 |
| 导出账单 | P2 | **砍掉** | 早期不需要 |
| 模型评分系统 | 扩展中 | **砍掉** | 用户不会主动给第三方模型评分 |

## 七、核心理念总结

> 这个产品的核心概念——"模型中立 + 引导式创作 + 角色一致性"——是站得住脚的。

**当前文档的核心不足不是缺内容，而是对用户创作心理节奏的理解不够深。**

八个步骤太长、角色一致性的模型差异没拆解、提示词体系有重叠、缺少预览闭环——这些问题叠加会让用户在实际使用中感到沉重而不是自由。

**最关键的一个建议**：先做一条 3 步极速路径（角色→剧情→一键出粗剪），让用户在 3 分钟内看到第一个作品，再用专业模式留住想要深度的用户。慢是一种奢侈，在产品冷启动阶段不能要求用户有耐心。
