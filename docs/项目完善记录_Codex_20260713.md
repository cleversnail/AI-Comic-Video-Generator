## 修复清单

### 1. 修复角色四视图/变体生成（严重 Bug）

**文件**：`apps/ai-video-api/src/modules/characters/characters.service.ts`

**问题**：`resolveImageKey` 返回 `apiKey: ''`（空字符串），导致四视图生成和变体生成两个对外暴露的端点完全无法工作。函数内还有死代码：`$queryRaw` 查询结果赋值后从未使用，`(this as any).configService` 在 service 上不存在。

**修复**：
- 注入 `ModelsService`（`CharactersModule` 已 import `ModelsModule`，依赖链已就绪）
- 删除整个 `resolveImageKey` 自实现逻辑，改为复用 `this.modelsService.resolveApiKey(userId, projectId, 'image')`
- 删除死代码（`$queryRaw`、`require('crypto')`、`(this as any).configService`）

**效果**：四视图生成和变体生成现在能正确解密 API Key 并调用图像模型。

### 2. 接线队列 Worker（核心功能缺失）

**文件**：
- 新增 `apps/ai-video-api/src/modules/queue/queue.processor.ts`
- 修改 `apps/ai-video-api/src/modules/queue/queue.module.ts`

**问题**：`QueueModule` 只注册了 `QueueService`（负责入队），从未实例化任何 Worker。`ImageWorker`/`VideoWorker` 类定义了但无人调用，且要求外部传入 `processFn`。结果：`GenerationsService.createTask` 把任务塞进 BullMQ 后没有任何消费者，整个异步生成链路（图像/视频/TTS）不工作。

**修复**：
- 新增 `QueueProcessor` 服务，实现 `OnModuleInit`/`OnModuleDestroy` 生命周期钩子
- 在 `onModuleInit` 中创建 BullMQ Worker，监听 `generation` 队列
- `processJob` 根据 capability 分发到对应 Adapter：
  - `image`：调 `ImageAdapter.generateImage`
  - `video`：调 `VideoAdapter.generateVideo`，若返回 `processing` 状态则轮询 `getVideoStatus` 直到完成
  - `tts`：调 `TTSAdapter.generateSpeech`
- 每个任务执行前标记 `processing`，成功后写入 `result` 并标记 `completed`，失败时写入 `error` 并标记 `failed`
- 若任务关联了 `shotId`，成功后自动回写 `shot.resultUrl` 和 `shot.status`
- `QueueModule` 新增 `imports: [PrismaModule, ModelsModule]`，注册 `QueueProcessor` 和 `AdapterFactory`

**效果**：异步生成任务现在能被真实消费，端到端生成链路打通。

### 3. 独立 API Key 加密密钥（安全加固）

**文件**：
- `apps/ai-video-api/src/modules/models/models.service.ts`
- `apps/ai-video-api/.env.example`

**问题**：`encryptApiKey`/`decryptApiKey` 使用 `JWT_SECRET` 派生 AES 密钥。JWT 密钥一旦泄露，攻击者既能伪造身份又能解密所有用户的 AI API Key。此外 `'your-jwt-secret'` 这个 fallback 默认值会导致生产环境忘配时静默使用弱密钥。

**修复**：
- 加密/解密函数改用独立的 `ENCRYPTION_KEY` 环境变量
- 去掉 fallback 默认值，若 `ENCRYPTION_KEY` 未配置则直接抛错（fail-fast，避免弱密钥）
- `.env.example` 新增 `ENCRYPTION_KEY` 配置项，注明必须与 `JWT_SECRET` 不同且至少 32 字符

**效果**：加密密钥与认证密钥隔离，消除密钥复用的安全风险。

### 4. 实现 FFmpeg 视频合成（核心闭环）

**文件**：`apps/ai-video-api/src/modules/compose/compose.service.ts`（重写）

**问题**：`composeProject` 只返回一个「合成计划已就绪」的占位字符串，从未调用 FFmpeg。核心价值闭环（故事 -> 视频）最后一公里没打通。

**修复**：用 `child_process.execFile` 调用系统 FFmpeg，实现完整的合成流程：

1. **素材校验**：过滤出有 `resultUrl` 的分镜，若无则报错
2. **FFmpeg 检测**：启动时检查 ffmpeg 是否可用
3. **逐镜构建**：
   - 有视频片段：re-encode 为 concat 兼容格式（libx264 + aac）
   - 仅有图片：`-loop 1` 生成静态画面视频，可选叠加音频轨
4. **拼接合成**：用 concat demuxer 合并所有片段
5. **字幕烧录**：若有字幕则生成 SRT 文件，用 `subtitles` 滤镜烧录
6. **上传存储**：合成产物上传 MinIO，返回可访问 URL
7. **状态回写**：更新 `project.status` 为 `composed`
8. **临时文件清理**：finally 块中递归删除工作目录

**效果**：合成导出从占位变为真实功能，最终视频上传到 MinIO 并返回 URL。

### 5. 依赖整理（工程卫生）

**文件**：
- `apps/ai-video-api/package.json`（后端）
- `apps/ai-video-web/package.json`（前端）

**问题**：后端 package.json 混入了 15+ 个前端依赖（`@radix-ui/*` x7、`@dnd-kit/*` x3、`framer-motion`、`react-hook-form`、`@hookform/resolvers`、`zustand`、`tailwind-merge`、`tailwindcss-animate`、`@tanstack/react-query`），而前端 package.json 反而缺少这些依赖，靠 pnpm hoisting 碰巧能跑。

**修复**：
- 后端移除所有前端依赖，只保留 NestJS + Prisma + BullMQ + MinIO 等后端依赖
- 前端补上 `@dnd-kit/*`、`@radix-ui/*`、`react-hook-form`、`@hookform/resolvers`、`zod`

**效果**：两个子包可独立安装，CI 严格隔离不会断裂，后端包体积减小。

### 6. Docker 一键部署配置

**新增文件**：
- `docker-compose.yml`（根目录）
- `apps/ai-video-api/Dockerfile`
- `apps/ai-video-web/Dockerfile`

**修改文件**：
- `apps/ai-video-web/next.config.mjs`（启用 `output: 'standalone'`）

**问题**：项目依赖 MySQL + Redis + MinIO + API + Web 五个服务，没有任何编排配置，新人无法快速启动。

**修复**：
- **docker-compose.yml**：编排 MySQL 8.0、Redis 7、MinIO、API、Web，全部带 healthcheck，API 依赖基础设施就绪后才启动
- **后端 Dockerfile**：多阶段构建（deps -> build -> prod），运行时镜像内置 ffmpeg
- **前端 Dockerfile**：多阶段构建，利用 Next.js standalone 输出减小镜像体积
- 环境变量通过 `${VAR:-default}` 支持 `.env` 覆盖

**使用方式**：
```bash
# 在项目根目录创建 .env 文件配置密钥
docker compose up -d
# 访问 http://localhost:3000（前端）、http://localhost:3001/api/docs（API 文档）
```

**效果**：一条 `docker compose up -d` 即可启动全套服务。

### 7. 分镜版本管理（数据安全）

**文件**：`apps/ai-video-api/src/modules/storyboard/storyboard.service.ts`

**问题**：`generateShots` 重生成时 `deleteMany` 物理删除所有旧分镜，无法回溯历史版本。另外有一处冗余的动态 `import('../../common/adapters/adapter.factory')` 遮蔽了已注入的 `this.adapterFactory`。

**修复**：
- 重生成时不再 `deleteMany`，改为将旧分镜标记为 `archived`（status 设为 `archived`，params 中写入 `archived: true` 和 `archivedAt`）
- `listShots` 过滤掉 `archived` 状态，只返回当前版本
- 删除冗余的动态 import

**效果**：分镜重生成不再丢失历史，用户可回溯对比不同版本。
